Cinderkeep - Code Style Rules

This project follows the code-consistency AI PDF style used by the class.
Use this as the team coding baseline.

Core Architecture
- Controllers only coordinate scene order, flow, and queues.
- Do not collect many unrelated Inspector references in one controller.
- Each actor GameObject owns the component that performs its own role.
- Examples:
  Movement belongs on the moving actor.
  Interaction belongs on the interactable actor.
  View belongs on the UI or visual actor.
  DropTarget belongs on the receiving actor.
- Prefer loose coupling.
- Use GetComponent, events, interfaces, or manager lookup before hard-wiring many references.
- If one controller needs 5 or more Inspector references, split the responsibility.

UI Rules
- Do not build normal UI with runtime new GameObject() calls.
- Create UI in the scene, with prefabs, or through editor tooling.
- Runtime code should activate, deactivate, bind, or update prepared UI objects.
- Reusable UI must become a prefab.

Data Rules
- Static data is planning data that does not change during play.
- Static data should be loaded through a data manager when the project grows.
- Instance data is play-state data that changes during play or save/load.
- Instance data should use Model naming, for example PlayerModel or ItemModel.
- Keep static data and instance data separate.

Naming Rules
- Private member fields use _prefix.
- Unity references are [SerializeField] private.
- Public access uses properties when needed.
- Methods start with verbs:
  LoadScene, SetProfile, RegisterInputEvents, ValidateSampleScene.
- Class names use PascalCase.
- One main class per file.

Component Rules
- Keep MonoBehaviours small.
- A component should have one clear reason to change.
- Avoid giant manager classes.
- Managers are allowed for common services or data ownership.
- Actors should not wait for a controller to do their own simple job.

Scene/Hierarchy Rules
- Important work objects must be visible in the Hierarchy.
- Name objects by role, not by vague labels like Object1.
- Shared work goes under WorkspaceRoot_Shared.
- Personal work goes under WorkRoot_Class_OwnerEnglishName.
- Placeholder objects are allowed early, but name their purpose clearly.

Current Project Examples
- CharacterSlot is owned by each character card object.
- CharacterSelectCanvas discovers CharacterSlot objects with GetComponentsInChildren.
- CharacterSceneLoadButton owns scene-loading behavior on the button object.
- MuckholdWorkspaceActor stores role data on each shared workspace placeholder.
- CharacterSelectSceneBuilder is editor-only tooling for scene generation and validation.

Review Checklist
- Does this component do one job?
- Is a controller holding too many references?
- Is this UI prepared in the scene or prefab instead of built at runtime?
- Is data separated from behavior when it starts growing?
- Can another team member understand the Hierarchy object name?
- Did this change touch only the intended system?
