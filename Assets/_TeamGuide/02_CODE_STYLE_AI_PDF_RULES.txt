Cinderkeep - Code Style Rules

This project follows the class code-consistency style.
Use this as the team coding baseline.

Core Architecture
- GameManager is the only singleton.
- GameManager owns the game flow and initialization order.
- Other managers prepare through Initialize instead of starting their own flow in Awake.
- Controllers coordinate flow only.
- Do not collect many unrelated Inspector references in one controller.
- Each actor GameObject owns the component that performs its own role.
- Prefer loose coupling through GetComponent, events, interfaces, or manager access.

Component Rules
- Keep MonoBehaviours small.
- A component should have one clear job.
- PlayerMovement moves the Player.
- PlayerView handles view-related work.
- UI components update prepared UI objects.
- Systems handle feature logic such as building, crafting, inventory, and combat.

UI Rules
- Do not build normal UI with runtime new GameObject calls.
- Create UI in scenes, prefabs, or editor tooling.
- Runtime UI code should activate, deactivate, bind, or update prepared objects.
- Reusable UI should become a prefab.

Data Rules
- Static Data is planning data that does not change during play.
- Static Data is loaded through GameDataManager when needed.
- Instance Data changes during play or save/load.
- Instance Data uses Model naming, such as PlayerModel or GameRunModel.
- Keep data and behavior separated when the code grows.

Naming Rules
- Private member fields use _prefix.
- Unity references are [SerializeField] private.
- Unity reference field names start with their type role, such as Button_StartGame.
- Public access uses properties or clear Get/Set methods when needed.
- Methods start with verbs, such as LoadScene, SetState, RegisterObject.
- Class names use PascalCase.
- One main class per file.

Folder Rules
- Official scripts go under Assets/Scripts by role.
- Experimental scripts go under Assets/_Sandbox/Scripts.
- Disabled scripts must use .cs.disabled.
- Editor-only scripts go under Assets/Scripts/Editor.

Review Checklist
- Does this component do one job?
- Is a controller holding too many references?
- Is this UI prepared in a scene or prefab instead of built at runtime?
- Is static data separated from runtime model data?
- Can another team member understand the object and file names?
- Did this change touch only the intended system?
