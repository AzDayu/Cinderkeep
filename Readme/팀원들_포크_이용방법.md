# 팀원들 포크 이용방법

이 문서는 Git, GitHub, Fork가 아직 익숙하지 않은 팀원이 Cinderkeep 프로젝트를 안전하게 작업하기 위한 팀 규칙입니다.

우리 팀은 9명이 함께 Unity 프로젝트를 만듭니다. 그래서 중요한 것은 Git 명령어를 많이 외우는 것이 아니라, 브랜치를 안전하게 나누고, 팀장이 확인한 뒤 합치는 것입니다.

## 한 줄 요약

```text
main에 직접 push하지 않습니다.
최신 develop에서 내 feature 브랜치를 만듭니다.
내 브랜치에서 작업합니다.
Push 후 Pull Request를 develop으로 만듭니다.
팀장이 확인하고 develop에 합칩니다.
팀장이 테스트 후 main에 반영합니다.
```

## 브랜치 구조

```text
feature/*
  -> develop
  -> main
```

- `main`: 발표, 제출, 공식 안정 버전입니다.
- `develop`: 팀원 작업을 모아 테스트하는 통합 브랜치입니다.
- `feature/*`: 팀원 개인 작업 브랜치입니다.
- `fix/*`: 버그 수정 브랜치입니다.
- `docs/*`: 문서 정리 브랜치입니다.
- `archive/*`: 예전 테스트용 브랜치, 폐기된 브랜치, 보관용 브랜치입니다.

새 작업은 `archive/*`에서 시작하지 않습니다.

## 팀장이 처음 한 번만 할 일

`develop` 브랜치가 GitHub에 아직 없다면 팀장이 먼저 만듭니다.

팀원이 이 작업을 직접 하지 않습니다.

```text
1. 최신 main을 받습니다.
2. main에서 develop 브랜치를 만듭니다.
3. develop 브랜치를 GitHub에 push합니다.
4. 그 다음부터 팀원에게 develop에서 feature 브랜치를 만들라고 안내합니다.
```

`develop` 브랜치가 생긴 뒤부터는 팀원 작업 기준이 `main`이 아니라 `develop`입니다.

## 작업 시작 전

Fork에서 매번 아래 순서를 지킵니다.

1. `Fetch`를 누릅니다.
2. `develop` 브랜치로 이동합니다.
3. `Pull`을 누릅니다.
4. 최신 `develop`에서 내 `feature` 브랜치를 만듭니다.
5. 내 브랜치로 이동한 상태에서 작업합니다.

Pull을 하지 않고 작업하면 다른 사람이 먼저 고친 내용이 내 컴퓨터에 없을 수 있습니다.

## 브랜치 이름

브랜치는 기능 단위로 만듭니다.

```text
feature/bard-hub-ui
feature/tanker-defense-ui
feature/mage-skill-effect
feature/archer-attack-test
feature/healer-item-test
fix/camera-error
docs/team-git-guide
```

## 작업 중 확인할 것

- 내가 어떤 브랜치에 있는지 항상 확인합니다.
- `main`에서 바로 작업하지 않습니다.
- `develop`에서 바로 작업하지 않습니다.
- 다른 사람의 개인 씬은 허락 없이 수정하지 않습니다.
- 공용 씬을 수정하기 전에는 팀장에게 말합니다.
- 유료 에셋은 `Assets/ThirdParty_PriAsset`에만 넣습니다.
- `Assets/ThirdParty_PriAsset` 안의 파일은 Git에 올리지 않습니다.

## Unity 씬 작업 규칙

Unity 프로젝트에서 가장 위험한 충돌은 씬 파일 충돌입니다.

아래 공용 씬은 여러 명이 동시에 수정하지 않습니다.

```text
Assets/Scenes/Main_Lobby.unity
Assets/Scenes/MainWorkspace/Muckhold_Workspace.unity
Assets/Scenes/MainWorkspace/MainWorkspaceRoom_ForBuild.unity
```

## Push와 Pull Request

Commit은 내 컴퓨터에 저장된 상태입니다. Push를 해야 GitHub에 올라가서 팀장이 볼 수 있습니다.

1. Fork 상단의 `Push`를 누릅니다.
2. 내 브랜치가 GitHub에 올라갔는지 확인합니다.
3. GitHub에서 Pull Request를 만듭니다.
4. Pull Request 대상은 `develop`으로 선택합니다.
5. 팀장에게 브랜치 이름과 작업 내용을 보냅니다.

팀장에게 보내는 예시:

```text
이름: 김성광
직업: 탱커
브랜치: feature/tanker-defense-ui
작업 내용: 탱커 개인 씬에 방어 UI 초안 추가
확인할 씬: Tanker_KimSeonggwang
특이사항: 아직 실제 체력 시스템과 연결하지 않음
```

## 팀장이 Merge하는 방식

팀원은 `main`에 직접 합치지 않습니다.

팀장 흐름:

1. 팀원 Pull Request를 확인합니다.
2. 팀원 브랜치를 가져옵니다.
3. Unity에서 프로젝트를 실행합니다.
4. Console 에러를 확인합니다.
5. 관련 씬을 열어봅니다.
6. 게임을 직접 플레이하거나 기능을 테스트합니다.
7. 문제가 없으면 `develop`에 Merge합니다.
8. `develop` 전체 테스트가 끝나면 `main`에 Merge합니다.

팀원이 "됐어요"라고 말해도 Unity 실행 전에는 Merge하지 않습니다.

## 충돌이 났을 때

충돌은 두 사람이 같은 파일의 같은 부분을 고쳤을 때 생깁니다. Unity 씬 파일에서 특히 자주 문제가 됩니다.

충돌이 나면 바로 멈춥니다.

하지 말 것:

- 아무거나 선택해서 덮어쓰기
- 잘 모르는데 Commit하기
- 잘 모르는데 Push하기
- Force Push하기
- 다른 사람 작업 지우기

해야 할 것:

```text
1. 충돌 난 파일 이름을 확인합니다.
2. 팀장에게 알립니다.
3. 왜 수정했는지 서로 확인합니다.
4. 팀장과 함께 어떤 버전을 남길지 결정합니다.
```

## 유료 에셋 주의

유료 에셋은 GitHub Public 레포지토리에 올리면 안 됩니다.

유료 에셋은 반드시 아래 폴더에만 넣습니다.

```text
Assets/ThirdParty_PriAsset
```

유료 에셋 공유는 GitHub가 아니라 Google Drive, Unity Package, 직접 파일 전달을 사용합니다.

## Git에 올리면 안 되는 폴더

Unity 프로젝트에서는 아래 폴더를 Git에 올리지 않습니다.

```text
Library
Temp
Obj
Logs
Build
Builds
```

## Fork에서 매일 볼 것

- `History`: 누가 언제 무엇을 커밋했는지 확인합니다.
- `Branches`: 살아있는 브랜치와 오래된 브랜치를 확인합니다.
- `Pull Requests`: 팀원이 올린 작업 요청을 확인합니다.

## 발표 전 백업

발표 전에는 팀장이 안정적인 상태에서 백업 브랜치를 만듭니다.

```text
release-v1
release-v2
```

문제가 생겨도 발표 가능한 버전으로 돌아가기 위한 안전장치입니다.

## 절대 하지 말아야 할 것

- `main`에 직접 push하기
- `develop`에 직접 push하기
- 팀장 확인 없이 Merge하기
- Force Push하기
- 유료 에셋을 GitHub에 올리기
- Library, Temp, Obj, Logs 폴더 올리기
- 다른 사람 개인 씬 마음대로 수정하기
- 공용 씬을 말 없이 수정하기
- 잘 모르는 상태로 충돌 해결하기

## 매일 작업 체크리스트

작업 시작:

```text
[ ] Fetch를 눌렀다.
[ ] develop으로 이동했다.
[ ] Pull을 눌렀다.
[ ] 최신 develop에서 내 feature 브랜치를 만들었다.
[ ] 현재 브랜치가 main/develop이 아닌 내 브랜치인지 확인했다.
```

작업 종료:

```text
[ ] 변경 파일을 확인했다.
[ ] 내가 고친 파일만 Stage했다.
[ ] 커밋 제목을 명확히 썼다.
[ ] Commit했다.
[ ] Push했다.
[ ] Pull Request를 develop으로 만들었다.
[ ] 팀장에게 브랜치 이름과 작업 내용을 보냈다.
```

## 최종 요약

`main`은 공식 안정 버전입니다.
`develop`은 팀 작업을 모아서 테스트하는 공간입니다.
`feature` 브랜치는 내 작업 공간입니다.

팀원은 최신 `develop`에서 `feature` 브랜치를 만들고, 자기 브랜치에서 작업한 뒤, Push하고 Pull Request를 만들어 팀장에게 확인받습니다.
